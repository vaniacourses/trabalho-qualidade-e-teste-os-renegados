# Análise de Qualidade ISO 25010 - Sistema WinxBank (Estado Alvo / Como Deveria Ser)

## 📊 Visão Geral

Este documento define as metas de qualidade para o sistema bancário **WinxBank** seguindo a norma ISO 25010. Atributos críticos para o domínio bancário possuem metas mais altas, enquanto outros são balanceados para viabilizar o projeto de forma eficiente.

---

## 🎯 Atributos de Qualidade e Medidas Esperadas

### 1. **ADEQUABILIDADE FUNCIONAL** - Escala Alvo: 5/5
**Medida Esperada:** O sistema deve possuir completude funcional das regras de negócio críticas e exatidão total (100%) em cálculos financeiros.
**Justificativa:** Em um sistema financeiro, a precisão matemática e o cumprimento das regras de negócio são inegociáveis. Erros aqui geram prejuízo direto e perda de clientes. É essencial que operações como saques, transferências e juros funcionem perfeitamente.

---

### 2. **SEGURANÇA** - Escala Alvo: 5/5
**Medida Esperada:** Criptografia de dados sensíveis, autenticação robusta (MFA) e conformidade com a LGPD.
**Justificativa:** No setor bancário, segurança é o pilar principal. Uma brecha pode significar falência, além de processos judiciais e sanções de órgãos reguladores. O sistema deve garantir confidencialidade, integridade, não-repúdio e rastreabilidade total das transações.

---

### 3. **CONFIABILIDADE** - Escala Alvo: 4/5
**Medida Esperada:** Disponibilidade de 99.9% (High Availability) com rotinas de backup regulares e tolerância a falhas nos serviços principais.
**Justificativa:** É financeiramente inviável (e tecnicamente excessivo para uma primeira versão) arquitetar para garantir 99.999% de disponibilidade. A meta de 4/5 (99.9%) garante que o sistema esteja operante na esmagadora maioria do tempo, com mecanismos de recuperação de dados suficientes para que não se perca o saldo de ninguém (garantia ACID), mas aceitando curtas janelas para manutenções programadas.

---

### 4. **MANUTENIBILIDADE** - Escala Alvo: 4/5
**Medida Esperada:** Código modular com cobertura de testes automatizados em torno de 80%, focados no domínio.
**Justificativa:** Buscar 100% de cobertura de testes gera *over-engineering* e atrasa a entrega de valor. Uma nota 4/5 garante que o código é testável e legível o suficiente para suportar refatorações e correções de bugs seguras.

---

### 5. **USABILIDADE** - Escala Alvo: 3/5
**Medida Esperada:** Interface gráfica limpa, clara e livre de ambiguidades, garantindo que o usuário consiga realizar suas tarefas.
**Justificativa:** A interface não precisa ter animações complexas, personalização extrema ou o design mais inovador do mercado. O foco principal é ser à prova de erros: evitar que o usuário transfira dinheiro para a conta errada por causa de um layout confuso, oferecendo feedback claro das ações.

---

### 6. **DESEMPENHO** - Escala Alvo: 3/5
**Medida Esperada:** Tempo de resposta aceitável (ex: entre 1 e 2 segundos) para operações normais e arquitetura capaz de lidar com a carga esperada para o curto/médio prazo.
**Justificativa:** Não precisamos construir um sistema capaz de suportar as transações simultâneas de toda a população global no dia 1. Atingir a escala 3/5 significa usar um banco de dados relacional com índices básicos.

---

### 7. **COMPATIBILIDADE** - Escala Alvo: 3/5
**Medida Esperada:** Utilizar APIs REST fundamentais utilizando padrões simples como JSON.
**Justificativa:** Um sistema moderno precisa conseguir se comunicar (ex: integração com o Bacen/PIX), mas não há necessidade de nascer com compatibilidade total para sistemas legados ou ser um hub avançado de Open Banking neste momento. Uma nota 3 atende o mercado atual de forma prática.

---

### 8. **PORTABILIDADE** - Escala Alvo: 3/5
**Medida Esperada:** Empacotamento em containers (Docker) e independência de caminhos locais (arquivos de configuração externos).
**Justificativa:** O sistema precisa rodar com previsibilidade fora da máquina do desenvolvedor. Contudo, não é necessário despender recursos para que o código seja multiplataforma. Rodar de forma confiável em um ecossistema conteinerizado atinge o objetivo.

---

## 📈 Resumo das Metas de Qualidade (Realismo de Engenharia)

| Atributo | Escala Alvo | Prioridade | Trade-off Justificado |
|----------|-------------|------------|-----------------------|
| **Adequabilidade Funcional** | 5/5 | Máxima | Sem concessões. Cálculo financeiro tem que ser exato. |
| **Segurança** | 5/5 | Máxima | Sem concessões. Vazamentos e falhas quebram a instituição. |
| **Confiabilidade** | 4/5 | Alta | Bom uptime (99.9%), mas sem o custo astronômico do 99.999%. |
| **Manutenibilidade** | 4/5 | Alta | Foco em testar o domínio bancário (80% cob), evitando over-engineering. |
| **Usabilidade** | 3/5 | Média | Interface funcional e à prova de erros, sem luxos estéticos. |
| **Desempenho** | 3/5 | Média | Uso de banco de dados padrão e índices, sem micro-otimizações. |
| **Compatibilidade** | 3/5 | Baixa | Apenas APIs REST fundamentais; integração básica. |
| **Portabilidade** | 3/5 | Baixa | Containerização básica (Docker) resolve o problema de implantação. |
